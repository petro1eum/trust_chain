Tool identity and code integrity

Most receipts sign what the agent claims.
Tool PKI proves which trusted tool produced it.

TrustChain Tool PKI binds a tool name, module, version, permissions, and source-code hash to a certificate. If the tool code changes, the certificate no longer matches. If the tool is compromised, it can be revoked.

1. Why Tool PKI matters

A signed tool call says a key signed some output. Tool PKI adds a stronger question: was that output produced by the certified tool implementation we expected?

Differentiator: TrustChain can sign the result and verify the tool identity/code hash. That is stronger than logging what the agent says it did.

Identity

Bind a tool to module, version, owner, organization, and issuer.

Integrity

Hash the tool source code and reject execution when code changes unexpectedly.

Governance

Attach permissions, expiration, trust level, revocation, and audit metadata.

5. OSS API

from trustchain.v2.certificate import ToolRegistry, trustchain_certified

registry = ToolRegistry()

def query_customer(customer_id: str) -> dict:
    return {"customer_id": customer_id, "balance": 100}

cert = registry.certify(
    query_customer,
    owner="Risk Engineering",
    organization="Acme Bank",
    permissions=["read:customer"],
)

assert registry.verify(query_customer)

@trustchain_certified(registry)
def guarded_tool(customer_id: str) -> dict:
    return query_customer(customer_id)

When the source changes, `registry.verify(...)` recomputes the hash and detects the mismatch. In strict mode, uncertified or tampered tools are rejected.

6. Receipts and registry

Tool certificates and agent certificates can be embedded in or referenced from `.tcreceipt` identity material. A verifier can then check both the signed output and the trusted tool identity.

EvidenceVerifier question answered
Signed envelopeWas this output changed after signing?
Public key / certificateWhich signer produced it?
Tool certificateWhich tool implementation produced it?
CRL / revocationWas that signer or tool revoked?
AnchorDoes the chain still match a previously published checkpoint?

8. Limits

  • Code hash checks do not prove the runtime or dependencies are uncompromised.
  • Built-ins, lambdas, generated functions, and native extensions may need manifest-based hashing.
  • A certificate does not prove a tool returns true data, only that it is the expected tool implementation.
  • High-assurance deployments should combine Tool PKI with SBOMs, KMS/HSM, dependency pinning, and runtime isolation.
