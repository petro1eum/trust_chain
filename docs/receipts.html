.tcreceipt
the evidence object TrustChain owns.

A .tcreceipt is a self-contained JSON proof that a tool output or agent response was signed by a known key. It is small enough to attach to tickets and strong enough to verify offline without calling the original agent.

1. Purpose

Logs are useful for debugging, but they are weak evidence. A receipt is different: it travels with the fact it proves. Anyone who receives it can verify the signature, inspect the tool identity, and decide whether the receipt is trusted in their own environment.

Sign

A real tool returns data and TrustChain signs the canonical envelope with Ed25519.

Carry

The signed envelope, public key, identity material, and witnesses are packaged as JSON.

Verify

A third party verifies the receipt offline and rejects tampering, replay, or wrong-key evidence.

2. Format

The current stable contract is format = "tcreceipt" and version = 1.

{
  "format": "tcreceipt",
  "version": 1,
  "issued_at": "2026-04-23T14:30:00Z",
  "envelope": {
    "tool_id": "sec_filing_lookup",
    "data": {"company": "Acme", "revenue_usd": 4812300000},
    "signature": "base64-ed25519-signature",
    "signature_id": "uuid",
    "timestamp": 1776954600.12,
    "nonce": "uuid",
    "parent_signature": null
  },
  "key": {
    "algorithm": "ed25519",
    "key_id": "producer-key-id",
    "public_key_b64": "base64-raw-ed25519-public-key"
  },
  "identity": null,
  "witnesses": null,
  "summary": {
    "tool_id": "sec_filing_lookup",
    "timestamp_iso": "2026-04-23T14:30:00Z",
    "signature_short": "abc12345..."
  }
}
FieldRequiredMeaning
formatYesMust be tcreceipt.
versionYesReceipt schema version. Current stable value: 1.
issued_atYesReceipt wrapping time in UTC. The signed event time is inside envelope.timestamp.
envelopeYesThe exact TrustChain signed response. This is what the signature covers.
keyYesVerifier key material and key id.
identityNoOptional certificate chain or registry-backed identity metadata.
witnessesNoOptional witness co-signatures or custody proofs.
summaryNoHuman-readable convenience fields. Verifiers must not rely on summary for cryptographic checks.

3. Verification model

Base verification is pure cryptography: rebuild the canonical envelope, verify the Ed25519 signature with key.public_key_b64, then apply optional policy checks like pinned key, maximum age, identity chain, and witnesses. 

from trustchain import Receipt

receipt = Receipt.load("acme_revenue.tcreceipt")
result = receipt.verify(expected_public_key_b64=trusted_public_key)
assert result.valid

Verifying without a pinned key proves internal consistency: the receipt was signed by the embedded key. Verifying with a pinned key or certificate chain proves it came from the expected signer.

4. Identity and custody

The OSS receipt can carry identity and witness material, but it does not require a hosted service. Private deployments can pin public keys directly. Public or cross-company deployments can use certificates, CRL snapshots, or a registry.

  • Local trust: distribute public keys internally and verify receipts offline.
  • PKIX trust: attach a certificate chain and verify against a known CA.
  • Witness trust: attach independent co-signatures for stronger custody evidence.
  • Enterprise trust: add KMS/HSM, retention, transparency service, and policy workflows outside the OSS core.

5. Standards export

The receipt is native. Standards exports are interoperability wrappers. This keeps TrustChain developer-friendly while allowing security teams to plug evidence into SCITT, W3C VC, and supply-chain systems.

TargetCommandOutput
SCITTtc standards export receipt.tcreceipt --format scittAIR-shaped JSON profile.
W3C VCtc standards export receipt.tcreceipt --format w3c-vcVC-shaped envelope embedding the native receipt.
in-tototc standards export receipt.tcreceipt --format intotoin-toto Statement v1.0 with TrustChain predicate.

6. Anchoring

A local hash chain detects edits inside the chain, but an attacker with full filesystem control could rewrite the whole chain. A portable anchor solves the evidence problem by storing the current HEAD and canonical chain digest somewhere the agent cannot change. 

tc anchor export -d .trustchain -o chain.anchor.json
# Store chain.anchor.json in Git, S3 Object Lock, a ticket, or a transparency log.
tc anchor verify chain.anchor.json -d .trustchain

OSS anchoring is intentionally simple: it exports a deterministic checkpoint. Enterprise custody can submit the same checkpoint to a transparency service, TSA, Rekor, or customer-owned evidence store.

7. CLI

tc receipt build signed_response.json --key pubkey.json -o result.tcreceipt
tc receipt show result.tcreceipt
tc receipt verify result.tcreceipt --pin <base64-public-key>

tc standards export result.tcreceipt --format scitt -o result.air.json
tc standards export result.tcreceipt --format w3c-vc -o result.vc.json
tc standards export result.tcreceipt --format intoto -o result.intoto.json

tc anchor export -d .trustchain -o chain.anchor.json
tc anchor verify chain.anchor.json -d .trustchain

8. Limits

TrustChain receipts are evidence, not magic. They prove the bytes were signed by a key. They do not prove the tool's real-world claim is true, that infrastructure was uncompromised, or that every event was captured unless policy requires fail-closed logging.

  • A valid receipt does not make bad data correct.
  • A receipt without identity pinning only proves consistency with its embedded key.
  • A local hash chain is tamper-evident, not globally tamper-proof, unless anchored externally.
  • Enterprise custody, retention, and legal evidence workflows depend on deployment architecture.
