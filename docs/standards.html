1. Honest positioning

TrustChain does not claim to invent digital signatures, hash chains, or receipts. Those primitives are deliberately boring and standard: Ed25519, SHA-256, deterministic JSON, optional PKIX, optional witness signatures.

The product claim is narrower and stronger: TrustChain turns those primitives into a working trust layer for AI tool outputs, agent execution chains, portable receipts, tool certificates, and compliance evidence.

The right comparison is not "blockchain for AI." It is closer to Git + Sigstore + PKI for AI execution evidence: sign the output, link the operation, preserve the receipt, verify offline, and export to the systems auditors already understand.

2. Functional map

Core

Signed facts

Tool output becomes a signed response with timestamp, nonce, optional parent signature, and offline verification.

Receipts

Portable proof

.tcreceipt bundles the envelope, key identity, certificate material, and witness data for third-party verification.

Chain

Execution history

Operations are linked into a tamper-evident trail stored in memory, files, Postgres, or an enterprise custody layer.

Tool PKI

Tool identity

Certificates identify tools and agents, making code tampering, key rotation, and revocation visible to verifiers.

Standards

Export layer

Receipts can be exported as SCITT-shaped AIR JSON, W3C VC envelopes, and in-toto Statements.

Custody

Enterprise evidence

KMS/HSM custody, transparency service, retention, policy, SSO, and audit workflows stay in the commercial platform.

3. Standards adapters

TrustChain keeps native receipts stable and adds export adapters instead of forcing every user into a single external standard. The adapters are deliberately lightweight in OSS and can be wrapped by Pro/Enterprise infrastructure.

AdapterOSS outputWhy it matters
trustchain.standards.scitt SCITT AIR-shaped JSON profile with content_hash, prev_chain_hash, chain_hash, sequence_number, agent_id. Prepares TrustChain evidence for SCITT/COSE transparency services without changing the native receipt.
trustchain.standards.w3c_vc W3C VC-shaped envelope embedding the native .tcreceipt as the credential subject. Lets identity and compliance systems handle receipts using issuer/subject/proof metadata.
trustchain.standards.intoto in-toto Statement v1.0 with a TrustChain predicate and SHA-256 subject digest. Connects AI tool evidence to Sigstore, cosign, DSSE, and supply-chain audit workflows. 
from trustchain.standards import to_scitt_air_json, to_w3c_vc, to_intoto_statement

air = to_scitt_air_json(signed_response, agent_id="agent:researcher", sequence_number=3)
vc = to_w3c_vc(receipt, issuer="did:web:trust-chain.ai", subject_id="did:example:agent")
statement = to_intoto_statement(signed_response)

The SCITT adapter is a JSON profile, not a COSE_Sign1 transparency receipt. The W3C adapter uses a TrustChain-specific proof type because the native signature covers the TrustChain envelope, not the outer VC JSON. This honesty keeps the OSS layer safe while leaving room for full enterprise custody.

4. .tcreceipt contract

A receipt is the portable object TrustChain owns. Everything else is an adapter. Verifiers should treat the native receipt as the source of truth and use standards exports as interoperability envelopes.

FieldPurpose
format / versionStable receipt contract: tcreceipt, currently version 1.
envelopeSigned response: tool id, data, timestamp, nonce, parent signature, metadata, certificate, TSA proof.
keyAlgorithm, key id, and public key required for offline verification.
identityOptional certificate material for PKIX or platform-backed identity.
witnessesOptional co-signatures from witness services or external custody.
summaryHuman-readable tool id, timestamp, and short signature preview.

5. Alternatives and overlap

The category is real. That is good: the market is validating cryptographic agent evidence. TrustChain should be positioned as a practical implementation and bridge, not as the only project in the field.

TrustChain should be sold to security and compliance teams as evidence infrastructure, not as model monitoring. It can help with automatic logging, traceability, audit integrity, provenance, and third-party verification.

7. What stays free and what stays paid

The OSS package must remain useful: signatures, receipts, offline verification, basic chain checks, CLI, SDKs, and standards export. Paid tiers should monetize operational scale, governance, custody, reporting, and enterprise identity.
